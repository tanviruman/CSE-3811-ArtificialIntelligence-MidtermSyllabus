Multiagent Environments — The Big Picture

So far all search problems had one agent acting alone. Now we deal with multiple agents — each agent must consider what other agents will do and how that affects its own outcome.

🤝 Co-operative Environments

Agents share a goal and help each other. Example: multiple robots cooperating to move a heavy object, taxi-driver cooperative systems.

⚔️ Competitive Environments

Agents' goals are in direct conflict. One agent's gain = another's loss. This gives rise to adversarial search problems — commonly called GAMES.

Adversarial Search: The unpredictability of other agents introduces contingencies — the optimal move depends on what the opponent does, and the opponent is actively trying to work against you.

♟️

Game Theory — What Kind of Games Does AI Study?

In AI, the most common games studied are deterministic, turn-taking, two-player, zero-sum games of perfect information. Classic examples: chess, checkers, Go, tic-tac-toe.

Breaking Down the Terminology

PropertyWhat it MeansExample
DeterministicNo randomness. Same action always leads to same result.Chess (no dice). Contrast: Backgammon (has dice → stochastic).
Turn-takingPlayers alternate moves — not simultaneous.Chess, checkers, tic-tac-toe.
Two-playerExactly two players: MAX and MIN.Chess (White vs Black).
Zero-sumOne player's gain = other player's loss. Net change in "wealth" is zero. Utility values at end are always equal and opposite.If MAX wins +1, MIN gets -1. Total = 0.
Perfect informationBoth players can see the entire game state at all times.Chess (all pieces visible). Contrast: Poker (hidden cards = imperfect).
💡

Zero-sum explained simply: If one player wins a chess game, the other necessarily loses. It is this OPPOSITION between utility functions that makes the situation adversarial. No draw = zero-sum. A draw can be 0,0 — still sums to 0.

The Two Players

MAX Player ▲

  • Wants to maximise the utility value
  • We (the AI agent) play as MAX
  • Moves at MAX nodes in the game tree
  • Picks the child with the highest value

MIN Player ▽

  • Wants to minimise the utility value
  • The opponent plays as MIN
  • Moves at MIN nodes in the game tree
  • Picks the child with the lowest value

Critical assumption in Minimax: The opponent (MIN) is assumed to play perfectly rationally — always choosing the move that hurts MAX the most. This is why it's called the "adversarial" assumption.

Game Tree — The Structure of a Game

A game tree is defined by:
• Initial state — the starting board configuration
• ACTIONS function — legal moves from any state
• RESULT function — what state results from each action
• TERMINAL-TEST — is this a leaf (game over)?
• UTILITY function — the value at terminal states (e.g., +1, -1, 0)

Key Properties of Game Trees

  • Root = current game state (it's MAX's turn to move)
  • MAX and MIN nodes alternate at each level
  • Leaf nodes (terminal states) have actual utility values (e.g., +1 = win, -1 = lose, 0 = draw)
  • Non-terminal nodes get their values assigned by Minimax (not given directly)
  • The game tree can be enormous — chess has ~1040 legal positions and ~10123 possible games

The Minimax Algorithm — Core Concept

The minimax value MINIMAX(n) of a node is the utility (for MAX) of being in that state, assuming both players play optimally. Think of it as: what is the best outcome MAX can guarantee if both sides play perfectly?

The Three Rules of Minimax

1

Terminal Node: MINIMAX(n) = UTILITY(n)

Just use the given leaf value. No calculation needed.

2

MAX Node: MINIMAX(n) = max over all successors s of MINIMAX(s)

Pick the HIGHEST value among all children. MAX wants the best outcome.

3

MIN Node: MINIMAX(n) = min over all successors s of MINIMAX(s)

Pick the LOWEST value among all children. MIN wants to hurt MAX.

MINIMAX(n) = UTILITY(n)                      if TERMINAL(n)
= max_s MINIMAX(s)                        if MAX node
= min_s MINIMAX(s)                        if MIN node

Simple Worked Example — Bottom-Up

Tree: Root=MAX, two MIN children (B, C), each with two leaf children.

B's leaves: 3, 12  |  C's leaves: 8, 2

// Step 1: Compute MIN nodes (B and C) from their leaves MINIMAX(B) = min(3, 12) = 3 // MIN picks LOWEST MINIMAX(C) = min(8, 2) = 2 // MIN picks LOWEST // Step 2: Compute MAX root from B and C's values MINIMAX(root) = max(3, 2) = 3 // MAX picks HIGHEST // Result: MAX should move to B (value=3) // If MAX moves to C, MIN will play leaf=2, giving MAX only 2 // If MAX moves to B, MIN will play leaf=3, giving MAX 3 (better!)

Properties of Minimax

PropertyValue / Explanation
Complete?Yes — if the game tree is finite
Optimal?Yes — against an optimal opponent
Time complexityO(bm) — exponential! b=branching factor, m=max depth
Space complexityO(bm) — DFS-style traversal
Against suboptimal opponent?Still wins, but may not exploit opponent's mistakes optimally
⚠️

Critical exam point: Minimax is optimal against a rational opponent. Against an irrational (random) opponent, minimax may actually perform suboptimally — you could exploit their mistakes better with a different strategy. But you'll never do WORSE than minimax guarantees.

⚙️

Minimax Algorithm — Pseudocode & How to Execute

// MINIMAX-DECISION: returns the best action for MAX at root function MINIMAX-DECISION(state): return argmax over actions a of MIN-VALUE(RESULT(state, a)) // MAX-VALUE: returns the best value MAX can achieve function MAX-VALUE(state): if TERMINAL-TEST(state): return UTILITY(state) v = -∞ for each action a in ACTIONS(state): v = max(v, MIN-VALUE(RESULT(state, a))) return v // MIN-VALUE: returns the best value MIN can achieve (worst for MAX) function MIN-VALUE(state): if TERMINAL-TEST(state): return UTILITY(state) v = +∞ for each action a in ACTIONS(state): v = min(v, MAX-VALUE(RESULT(state, a))) return v

How to Execute Minimax on a Game Tree — Step by Step

  1. Identify terminal nodes (leaves). Their values are given.
  2. Work bottom-up. Start from the deepest non-terminal level.
  3. At MAX nodes: take the maximum of all children's values.
  4. At MIN nodes: take the minimum of all children's values.
  5. Propagate up until you reach the root.
  6. At the root (MAX): choose the action that leads to the child with the highest value — that's the optimal move.

Order of evaluation: Minimax uses DFS (depth-first). It goes to the leftmost leaf first, then backtracks. Unless told otherwise, assume left-to-right traversal.

⏱️

Resource Limits — Why We Can't Always Search to Leaves

The Problem: Real games like chess have ~10123 possible game sequences. Even at 10K nodes/second with 100 seconds per move = 1 million nodes. Alpha-beta reaches ~depth 8. We cannot search to the leaves in realistic games.

Solution: Depth-Limited Search + Evaluation Function

Depth-Limited Search

Stop searching after a fixed depth (e.g., 8 plies). Don't expand deeper.

Tradeoff: Faster, but may miss threats/opportunities beyond depth limit.

Evaluation Function

Replace UTILITY at cut-off nodes with an evaluation function that estimates how good the position is — without searching further.

Example for chess: count material (queen=9, rook=5, etc.), board control, king safety.

Effects of Using Evaluation Functions

  • Guarantee of optimal play is gone — evaluation functions are imperfect approximations
  • Deeper search compensates — the deeper the evaluation function is buried, the less its quality matters (the tree itself corrects errors)
  • Tradeoff: More complex evaluation function = more accurate but slower; simpler = faster but less accurate

Key exam insight: Why use evaluation functions at all? Because O(bm) is too expensive for large m. With depth limit and eval function, we sacrifice guaranteed optimality for practical speed.

✂️

Alpha-Beta Pruning — Skip Irrelevant Branches

Core idea: Minimax must examine every node — O(bm). Alpha-beta prunes branches that cannot possibly affect the root's decision. Same result as minimax, but computed faster. Best case: O(bm/2) — as if the tree has HALF the depth!

The Two Values

α (alpha)

The best value MAX has found so far along the current path. Initially −∞.

Updated at MAX nodes when a better child is found.

"The floor MAX won't go below"

β (beta)

The best value MIN has found so far along the current path. Initially +∞.

Updated at MIN nodes when a better (lower) child is found.

"The ceiling MIN won't exceed"

The Pruning Condition

PRUNE when: α ≥ β

In plain English: If MAX already has a guaranteed path worth α, and MIN is currently offering at most β ≤ α, then MIN will never let MAX take this path. MAX won't come here. Skip remaining children!

Pruning Rules — Which Nodes to Cut

Node TypeWhen to PruneWhy
MIN nodeIf current node's value v ≤ α (parent's MAX guarantee)MAX already has something better (α) elsewhere. MAX won't choose this MIN node since it gives ≤ α. Prune remaining children of this MIN node.
MAX nodeIf current node's value v ≥ β (parent's MIN guarantee)MIN already has something better (β ≤ v) elsewhere. MIN won't let this MAX node be reached. Prune remaining children of this MAX node.
💡

Memory trick: "Alpha is MAX's safety net (updated at MAX nodes). Beta is MIN's ceiling (updated at MIN nodes). Cut when safety net ≥ ceiling — they've overlapped and this branch is irrelevant."

Alpha-Beta Algorithm

// Alpha-Beta Search — returns value at root function ALPHA-BETA(node, α, β, isMax): if TERMINAL(node): return UTILITY(node) if isMax: // MAX node v = -∞ for each child c: v = max(v, ALPHA-BETA(c, α, β, false)) α = max(α, v) // update alpha if α ≥ β: BREAK // β-cutoff! Prune remaining siblings return v else: // MIN node v = +∞ for each child c: v = min(v, ALPHA-BETA(c, α, β, true)) β = min(β, v) // update beta if α ≥ β: BREAK // α-cutoff! Prune remaining siblings return v // Call at root: ALPHA-BETA(root, -∞, +∞, true)

Step-by-Step Rules for Exams

  1. Start at root with α=−∞, β=+∞
  2. Pass (α, β) down to children — children inherit parent's α, β
  3. At MAX node: update α = max(α, child_value) after each child. If α ≥ β → prune remaining children.
  4. At MIN node: update β = min(β, child_value) after each child. If α ≥ β → prune remaining children.
  5. Propagate values up same as minimax — MAX takes max, MIN takes min.
  6. Final root value is identical to regular minimax.

Key fact: Alpha-beta pruning does NOT change the minimax value. The root still gets the exact same answer. It only SKIPS the evaluation of branches that can't affect the result. This is why it's safe to prune.

General Alpha-Beta for Left→Right Traversal

▶ KEY STRATEGY FOR EXAM
  1. Identify all terminal leaf values first (given at the bottom).
  2. Group them by their parent nodes (MAX nodes at the deepest non-leaf level).
  3. Compute those MAX nodes first.
  4. Then compute MIN nodes above them.
  5. Continue up to root.
  6. For alpha-beta: after evaluating the first MIN child of root, you have α. Use it to prune subsequent MIN children if they start returning values ≤ α.

Critical pattern: Whenever a MIN node's value drops BELOW the current α (root's best guarantee), PRUNE all remaining children of that MIN node. The root already has something better.

Summer 2025 Midterm — Q6 Alpha-Beta (Right-to-Left!)

Summer 2025 Q6: "The children of each node must be visited from right to left during the minimax search." Find values for a, b, c, d such that highlighted branches are pruned. Indicate utility values at internal nodes.

⚠️ This is the trickiest exam variant! Right-to-left traversal means you evaluate the RIGHTMOST children first. This changes which branches get pruned — but the final root value remains the same as regular minimax!

Right-to-Left Alpha-Beta — How it Changes Things

Left-to-Right (normal)

Evaluate leftmost child first → sets initial α or β → prunes right siblings if condition met.

Same subtrees are evaluated, different ones are pruned compared to R→L.

Right-to-Left (Summer 2025)

Evaluate rightmost child first → sets initial α or β → prunes LEFT siblings if condition met.

The pruned nodes are DIFFERENT but the root value is the SAME.

Summer 2025 Tree (leaf values from PDF): Known leaves = 4, 1, 3, b?, 1, 2, a?

▶ GENERAL METHOD FOR "FIND RANGES OF a,b,c,d"
// Structure from PDF: Root=MAX, 3 MIN children, // Each MIN has 2 MAX grandchildren, // Some leaves are unknown (a, b, c, d) // Strategy: The highlighted (pruned) branches tell you what MUST be true // for the pruning condition α ≥ β to hold at those specific nodes. // Example: If branch with leaf 'a' is pruned: // That means at the MIN node containing 'a', β ≤ α at the time 'a' is reached. // So: whatever β is at that point must be ≤ current α. // This gives you an INEQUALITY that defines the range of 'a'. // Step-by-step: 1. Compute all KNOWN internal node values first (bottom-up, right-to-left) 2. Find what α and β values are at the pruning points 3. For unknown leaf 'a' to cause pruning: derive the inequality 4. Express as range: a ∈ [x, +∞) or a ∈ (−∞, y] etc.

For the Summer 2025 tree specifically (from image): Known leaves: 4, 1, 3, 1, 2. Unknowns: a, b, c, d. Right branch has known leaf 4 and unknown a. Working right-to-left, evaluate right subtrees first to set α before reaching the left branches where pruning may occur.

💡

Key insight for "range" questions: A branch is pruned when visiting it is redundant — when even the best possible value it could return wouldn't change the decision. Unknown variables define the "threshold" of this redundancy. State the inequality clearly, e.g.: "a ≤ 3" (because if a ≤ 3, the MIN node value ≤ 3 < α=4, causing α-cutoff).

Q5b — Does Minimax Always Provide an Optimal Solution?

▶ ANSWER (high-value exam question)

No, Minimax does NOT always provide an optimal solution. Here's why and an example:

When Minimax IS optimal:

  • Against a perfectly rational opponent
  • When the full game tree is searched to terminal nodes
  • When utility values are accurate

When Minimax is NOT optimal:

  • Against an irrational/random opponent (you may miss exploiting their mistakes)
  • With depth-limited search + evaluation function (approximation, not exact)
  • In stochastic games (random outcomes, minimax doesn't account for chance nodes)

Counterexample: If the opponent always makes the worst possible move (very irrational), a minimax agent might still win but by a smaller margin. A different strategy designed to exploit the irrational opponent could achieve a higher expected utility.
