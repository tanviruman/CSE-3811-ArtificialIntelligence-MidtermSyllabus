Why Do We Need UCS?

Problem with BFS: BFS finds the path with the fewest steps, but NOT the cheapest path when edges have different costs. Example: going 3 steps each costing 1 is NOT always cheaper than going 2 steps each costing 10!

BFS limitation: BFS is only optimal when ALL edge costs are equal (like all = 1). In real problems (road distances, network bandwidth, time costs), edges have different weights.

UCS solution: Instead of expanding by depth (number of steps), expand by total path cost. Always process the cheapest accumulated path first. This guarantees the optimal (cheapest) solution.

Intuition — Think of Dijkstra's Algorithm

UCS is essentially Dijkstra's shortest path algorithm applied to AI search. If you've heard of Dijkstra, UCS is the same thing! It "explores outward" from the start in order of increasing cost, like concentric cost-circles expanding.

How UCS Works

Core Mechanism: UCS uses a Priority Queue sorted by cumulative path cost g(n). The node with the smallest g(n) is always expanded next. Goal is checked when a node is popped, not when it's added to the queue.

The Priority Queue — Min-Heap

A Priority Queue automatically keeps elements sorted. The SMALLEST element is always at the front. This is typically implemented as a Min-Heap.

// From your notes: Priority Queue example Queue: [A, B, C, D, E] [4, 7, 2, 5, 1] ← costs Priority Queue sorts by cost: PQ: [E(1), C(2), A(4), D(5), B(7)] ↑ Pop this first (smallest cost) // Your notes: "Minimum value বেছে নেবে" (will select minimum value) // Implement using Min Heap (also called Min-Priority Queue)

Backward Cost g(n)

g(n) = the total cost of the path from the start node to node n. This is the "backward cost" — how much we've already spent getting here. UCS always expands the node with the smallest g(n).

// Example: Path S → A → C has backward cost: g(S) = 0 // start node, zero cost g(A) = cost(S,A) = 6 g(C) = g(A) + cost(A,C) = 6 + 5 = 11

UCS Properties

PropertyResultExplanation
Completeness✓ YESWill always find a solution, provided all edge costs are positive (> 0) and branching factor is finite.
Optimality✓ YES (always!)ALWAYS finds the cheapest path, regardless of edge costs. This is UCS's defining advantage over BFS.
Time ComplexityO(bC*/ε)C* = optimal cost, ε = minimum edge cost. Effectively the "cost depth" of the solution.
Space ComplexityO(bC*/ε)Stores the frontier — can be large if there are many cheap paths to explore before finding goal.

Why UCS is always optimal: When UCS pops a node n from the priority queue, it has been reached via the cheapest possible path from start. Any other path to n would cost at least as much (otherwise it would have been popped earlier). Therefore, the first time the goal is popped, it's via the cheapest path.

⚠️

CRITICAL: In UCS, check if a node is the goal when it is POPPED from the queue, NOT when it is added to the queue. Adding to queue doesn't mean you've found it optimally yet — another cheaper path might arrive later!

UCS Algorithm — Step by Step

1

Add the start node to the Priority Queue with cost g = 0

2

Pop the node with the LOWEST total cost g(n) from the Priority Queue

3

Check: Is this node the GOAL? If YES → done! Return the path and cost.

4

If this node has been visited before → SKIP it (already explored via a cheaper path)

5

Mark this node as visited/explored

6

For each neighbour: calculate new_cost = g(current) + edge_cost(current, neighbour)

7

Add each unvisited neighbour to the Priority Queue with its new_cost

8

Repeat from step 2 until queue is empty (no solution exists) or goal found

💡

Tie-breaking: When two nodes have equal cost, exam questions usually say "alphabetical order" — process alphabetically first (A before B before C, etc.).

// UCS Pseudocode (conceptual): function UCS(start, goal): frontier = PriorityQueue() frontier.add(start, priority=0) explored = {} g = {start: 0} while frontier is not empty: node = frontier.pop_min() // pop lowest cost if node == goal: return path // CHECK AT POP! if node in explored: continue explored.add(node) for each neighbor, cost in node.successors: new_g = g[node] + cost if neighbor not in explored: frontier.add(neighbor, priority=new_g) g[neighbor] = new_g return FAILURE

Common Mistakes to Avoid in Exams

Mistake 1: Checking goal when ADDED to queue (not when POPPED)

Wrong: "Add G to queue → it's the goal → return immediately"
Right: Must wait until G is POPPED from priority queue. Another cheaper path to G might still be in queue!

Mistake 2: Not skipping already-explored nodes

When a node is popped from the PQ, first check if it's already in the "explored" set. If yes, SKIP IT. Otherwise you'll process it multiple times.

Mistake 3: Confusing UCS with BFS on weighted graphs

BFS is NOT optimal on weighted graphs! Always use UCS when edge costs differ. BFS = optimal for unweighted (equal cost) graphs only.

Mistake 4: Wrong tie-breaking

When two nodes have equal cost in PQ, break ties alphabetically (or as specified in the question). This affects which path is found but not the optimal cost.

🏆

UCS guarantee: The first time a node is POPPED from the priority queue, it has been reached via the globally cheapest path. This is why UCS is always optimal when costs are positive!
