Why Local Search? Classical vs Local

Classical Search (A*, BFS, UCS)

  • Keeps entire path in memory
  • Returns a path from start to goal
  • Systematic — explores the space
  • Memory hungry (O(bˢ) or worse)
  • The PATH is the solution

Local Search (HC, SA)

  • Only current state in memory (constant!)
  • Moves to neighbours directly
  • NOT systematic — no path retained
  • Memory efficient — O(1) space
  • The GOAL STATE is the solution

When to use local search? When only the final solution matters — not HOW you got there. Examples: 8-queens (just need a valid queen placement), circuit layout, scheduling, protein folding.

From your notes (Class 7): "Local Search → Hill-climbing → Simulated Annealing." HC is for Maximisation (dark cloud). A*-Search is for Minimisation (light cloud).

State-Space Landscape

Imagine a hilly landscape where:

  • Position = a state (current configuration)
  • Elevation = the value of the objective function / utility value
  • Goal for maximisation = find the highest peak (global maximum)
  • Goal for minimisation = find the lowest valley (global minimum)

From your notes: "Highest value → Find করা" (find the highest value = target state). "Maximum value → পাওয়া" (get the maximum value).

Hill Climbing Search

Core Idea: Continuously move in the direction of increasing value (uphill). Stop when no neighbour has a higher value. Like being blindfolded on a mountain — you can only feel which direction is uphill.

From your notes: "Greedy local search" because it always takes the best immediate step without thinking ahead. Does NOT maintain a search tree.

Hill Climbing Algorithm

// Steepest Ascent Hill Climbing (maximisation) current = start_state loop: neighbours = all_successors(current) best_neighbour = max(neighbours, by objective_value) if value(best_neighbour) ≤ value(current): return current // stuck! no improvement possible current = best_neighbour // move uphill

8-Queens Example

State: 8 queens on board, one per column. Successors: move any queen to any other square in same column → 8×7=56 successors. Heuristic h = number of attacking queen pairs (want to minimise this → h=0 is perfect). Hill climbing reduces h step by step until stuck.

Performance: Gets stuck 86% of the time! Only solves 14% of random initial configurations. Takes about 4 steps when it succeeds, 3 when it fails.

⚠️

Hill Climbing Problems & Solutions

Problem 1: Gets Stuck at Local Maxima

HC stops at any peak, even if it's not the global maximum. Every neighbour is lower, so HC thinks it's done — but it's just at a local maximum, not the best possible solution.

Problem 2: Can't Navigate Ridges

A ridge has steep sides but a narrow top. Moving in any direction from a ridge-point goes downhill (except along the ridge). HC gets trapped and can't make progress.

Problem 3: Gets Trapped on Plateaux

All neighbours equal the current value → HC sees no improvement → stops. Might be a flat local max (stuck forever) or a shoulder (could continue if it allowed sideways moves).

Solutions (Variants of Hill Climbing)

VariantHow It WorksSolvesProblem
Sideways MovesAllow moving to a neighbour with EQUAL value (not just better). Stop after N consecutive sideways moves.Plateaux (shoulders)May loop forever without the limit N. 8-queens: success rises from 14% to 94%!
Random RestartRun HC k times from different random starting points. Return the best result found.Local maximaEventually finds global optimum with enough restarts. Each run is independent.
Stochastic HCInstead of always taking the BEST uphill step, randomly choose an uphill step weighted by how much better it is.RidgesMore exploration but slower convergence.
Simulated AnnealingSometimes accepts downhill moves with a probability that decreases over time.All of the above!More complex — requires cooling schedule tuning.

Q4a from Assignment: Sideways Moves in 8-Queens

▶ ANSWER

Allowing sideways moves means accepting moves to states with equal objective value (not strictly better). This helps escape plateaux (flat areas). Without sideways moves, HC solves 8-queens only 14% of the time. With sideways moves (capped at ~100), success rises to ~94%. The tradeoff: it's slower per successful run because of extra sideways exploration.

Q4b: Local Beam Search vs Random Restart

▶ KEY DIFFERENCE

Random Restart (k runs)

Run k completely independent HC searches from random starts. No information passes between runs. After all k, pick the best answer.

Local Beam Search (k states)

Maintain k states simultaneously. At each step, generate ALL successors of all k states, keep best k. States communicate — promising states generate more offspring. Much more information sharing!

Key insight: Beam search is NOT k independent searches. If k=1, beam search = regular hill climbing. If k=∞, beam search = BFS!

Simulated Annealing — The Escape Artist

Inspiration: Annealing in metallurgy — heat metal then cool slowly → atoms settle into low-energy crystalline structure. SA mimics this: start "hot" (accept many bad moves), gradually "cool" (accept fewer bad moves).

Core Idea: Like hill climbing, but occasionally accepts downhill (bad) moves with a probability that decreases as temperature T decreases. This allows escaping local optima!

SA vs Hill Climbing

Hill Climbing

✅ Always accept better moves

❌ Never accept worse moves

Result: Gets stuck at local optima

Simulated Annealing

✅ Always accept better moves

⚡ Sometimes accept worse moves (probability depends on T)

Result: Can escape local optima!

🧠

From your notes (page 28): "Threshold 1 means all accepted. Threshold 0 means worst — all rejected. Basic Hill Climbing." When T=∞, prob=1 (all accepted). When T=0, prob=0 (none accepted, like HC).

📐

SA Formula — The Most Important Part!

For MAXIMISATION Problems

ΔE = E(neighbour) − E(current). If ΔE > 0 (neighbour is BETTER): always accept (move there). If ΔE < 0 (neighbour is WORSE): accept with probability eΔE/T.

P(accept bad move) = eΔE/T where ΔE = E(neighbor) - E(current)

Accept the bad move if: Random Number (RNG) < eΔE/T

For MINIMISATION Problems

ΔE = E(neighbour) − E(current). If ΔE < 0 (neighbour is BETTER = lower): always accept. If ΔE > 0 (neighbour is WORSE = higher): accept with probability e-ΔE/T.

P(accept bad move) = e-ΔE/T (note negative sign for minimisation!)

From your notes (page 27): Table shows "Maximisation: ΔE=E₂-E₁, prob = eΔE/T" and "Minimisation: prob = e-ΔE/T"

Effect of Temperature T

TemperatureEffect on eΔE/TBehaviour
T → ∞ (very high)eΔE/T → e⁰ = 1All moves accepted! Pure random walk. Your notes: "T=∞ → High Temperature → Prob = e^(ΔE/∞) = e⁰ = 1 → Threshold value 1 means all accepted."
T is moderate0 < eΔE/T < 1Bad moves accepted with some probability — balance exploration and exploitation.
T → 0 (very low/cold)eΔE/T → 0Bad moves almost never accepted. Behaves like pure hill climbing. Your notes: "Threshold 0 means worst — all rejected = Basic Hill Climbing."
💡

Memory trick: High T = Hot = Wild = random walks (explore). Low T = Cold = Calm = only accepts good moves (exploit). SA starts hot and cools down over time.

SA Algorithm Steps

// SA Algorithm for Maximisation current = start_state T = initial_temperature for t = 0, 1, 2, ...: T = temperature_schedule(t) // T decreases over time neighbour = random_neighbour(current) ΔE = E(neighbour) - E(current) if ΔE > 0: // neighbour is BETTER current = neighbour // ALWAYS accept! else: // neighbour is WORSE prob = e^(ΔE / T) // probability to accept bad move RNG = random(0, 1) if RNG < prob: // "second chance" current = neighbour // accept bad move! // else: stay at current (reject bad move) // NOTE: "Once Accepted there is no Check" — from your notes! // Once you move to a neighbour, don't re-evaluate it against current
⚠️

From your notes (page 27): "Once Accepted there is no Check" — after accepting a neighbour (good or bad), you move there unconditionally. Don't second-guess it! And the RNG for the NEXT timestep is calculated fresh.
