Master Pitfall List — 25 Exam Traps Explained

These are the most common mistakes students make. Each one has been seen in actual exam answers. Read carefully — these are where marks are lost.

PEAS Pitfalls

PITFALL #1 · PEAS

Confusing "Environment" with "Task" — Giving Abstract Answers

❌ WRONG

Environment: "The place where the robot works"

✅ RIGHT

Environment = specific physical/digital world the agent operates in. For hospital robot: "Corridors, patient rooms, pharmacy, intersections, other hospital staff, blocked paths, battery levels." Be specific and concrete.

PITFALL #2 · PEAS

Forgetting Performance is a MEASURE — Not just "doing a good job"

❌ WRONG

Performance: "The robot should work well and deliver medicine"

✅ RIGHT

Performance: "+10 for each medicine delivered on time, -5 for wrong room, -1 per battery unit used, -50 for collision". Performance = a measurable score or objective function that tells us how good the agent is doing.

PITFALL #3 · PEAS

Mixing Up Sensors and Actuators

❌ WRONG

Sensors: Wheels. Actuators: Camera.

✅ RIGHT

Sensors = INPUT devices (camera, microphone, GPS, temperature sensor — things that SENSE). Actuators = OUTPUT devices (wheels, motors, robotic arm, display, speaker — things that ACT). Always ask: does this device gather info (sensor) or cause physical change (actuator)?

Search Algorithm Pitfalls

PITFALL #4 · SEARCH

BFS is NOT Always Optimal

❌ WRONG BELIEF

"BFS always finds the cheapest path, so I'll use it for weighted graphs."

✅ TRUTH

BFS finds the path with the fewest edges/hops, NOT the cheapest cost. In the assignment graph (S→A:1, S→C:10, A→B:5, B→C:2, C→G:1), BFS returns S→C→G (cost 11, fewest hops=2) but UCS returns S→A→B→C→G (cost 9, more hops but cheaper). BFS is only optimal when all edge costs are equal.

PITFALL #5 · SEARCH

DFS Can Return Non-Optimal Paths (But May Coincidentally Find Optimal)

❌ WRONG

"DFS returned the cheapest path so it's optimal."

✅ RIGHT

DFS is not guaranteed to be optimal. In the assignment graph, DFS happens to return S→A→B→C→G (cost=9, which is optimal), but this is coincidental. DFS doesn't check costs — it just goes deep first. In a different graph, DFS could easily find a longer, more expensive path.

PITFALL #6 · SEARCH

UCS and Dijkstra — They're the Same Thing

UCS (Uniform Cost Search) is essentially Dijkstra's algorithm. Both expand nodes in order of cumulative cost g(n). Both are complete and optimal when edge costs > 0. In exams, always use a priority queue, always pop the minimum cost node, and always skip already-visited nodes (graph search).

PITFALL #7 · SEARCH

Forgetting Tie-Breaking in BFS/DFS/UCS

❌ WRONG

Ignoring the tie-breaking rule stated in the question.

✅ RIGHT

Exam questions almost always say "break ties alphabetically" or "Up>Down>Right>Left". This completely changes which node is visited first and which path is returned. Always read and apply the tie-breaking rule FIRST before tracing. In BFS: among equal-priority nodes, sort alphabetically. In UCS: among equal-cost nodes, sort alphabetically.

PITFALL #8 · SEARCH

Graph Search vs Tree Search — Handling Visited Nodes

Tree Search

Never marks nodes as visited. Can visit the same node multiple times via different paths. More memory. Used for A* Tree Search (needed when h is only admissible, not consistent).

Graph Search

Maintains a "closed" set of visited nodes. Never re-expands them. More efficient. Required when the question says "graph search". Used for BFS/DFS/UCS typically.

When a question says "if a node is not expanded twice" = they mean graph search. When they ask you to "show the tree" = they want tree search. Most exam questions specify — read carefully.

PITFALL #9 · SEARCH

DFS/BFS on STRING/PASSWORD Problems — What Each Returns

// Spring 2025 Q2: Strings of X,Y,Z. Cost X=1, Y=2, Z=3. // Correct passwords: XXXZZZ, XYYZZ, YXYXY, YZXYXZY, ZYXZ, ZYXZY DFS → goes DEEPEST first via ALPHABETICAL path (X before Y before Z) → goes X→XX→XXX→...→XXXXXXX→...backtrack→...→XXXZZZ ✅ → Returns: XXXZZZ (deepest alphabetically-first path that matches) BFS → explores ALL strings of length 1, then length 2, then length 3... → finds SHORTEST matching string → ZYXZ (length 4) → Returns: ZYXZ (shallowest = shortest password) UCS → expands by CUMULATIVE COST (X=1, Y=2, Z=3) → finds CHEAPEST matching string first → YXYXY: 2+1+2+1+2 = 8 ← cheapest → Returns: YXYXY (minimum total cost)

Heuristic & A* Pitfalls

PITFALL #11 · HEURISTIC

Admissible ≠ Consistent — Critical Distinction!

Admissible: h(n) ≤ h*(n) for all n. Never overestimates. Required for A* Tree Search to be optimal.

Consistent (Monotone): h(n) ≤ c(n,n') + h(n') for all edges n→n'. Required for A* Graph Search to be optimal.

Relationship: Consistent → Admissible. But Admissible does NOT imply Consistent.

In the A* graph (S,A,B,C,D,G) from the assignment: h(S)=6. c(S→C)=2, h(C)=3. Check: 6 ≤ 2+3=5? NO! 6 > 5 → Inconsistent! So A* Graph Search is NOT guaranteed optimal here. A* Tree Search IS optimal (h is admissible).

PITFALL #12 · HEURISTIC

How to Check Admissibility — The Correct Method

▶ CORRECT PROCEDURE
  1. For each node n, find the actual shortest path cost from n to goal (h*(n)). Use UCS or just trace all paths.
  2. Compare h(n) with h*(n). If h(n) > h*(n) for ANY node → NOT admissible.
  3. If you need to fix it: set h(n) = h*(n) or any value ≤ h*(n).

Common values to check from exam graphs: h(goal) must be 0. If h(goal) ≠ 0, immediately NOT admissible. Also check nodes adjacent to goal — they must have h ≤ edge cost to goal.

PITFALL #13 · HEURISTIC

Heuristic Combinations — Which Are Admissible?

FormulaAdmissible?Why (memorise this)
max(h1, h2, h3) — all admissible✅ YESmax of admissible ≤ true cost (since each hi ≤ h*)
min(h1, h2, h4) — h4 inadmissible✅ YESmin ≤ each individual value → ≤ admissible ones → ≤ h*
max(h1, h2, h3, h4) — h4 inadmissible❌ NOh4 can exceed h*, max might take h4's value
h1² + h2 + h3❌ NOSquaring makes h1² > h* even if h1 ≤ h*

average(h1,h2,h4) — h4 inadmissible❌ NOh4 can be huge, pulls average above h*

💡

Rule of thumb: If h4 (inadmissible) appears in a MAX or SUM operation → likely NOT admissible. If h4 appears ONLY in a MIN operation with admissible heuristics → still admissible (min filters it out).

PITFALL #14 · HEURISTIC

A* Tree Search vs A* Graph Search — Different Results Possible!

▶ KEY SCENARIO WHERE THEY DIFFER

When h is admissible but NOT consistent, A* Graph Search may close a node via a suboptimal path, then find a better path to the same node — but ignore it because the node is already closed. This gives a non-optimal result.

// Example from assignment/midterm: // h(S)=6, but c(S→C)=2, h(C)=3 → inconsistent: 6 > 5 A* TREE Search: Opens: S(f=6), C(f=5), D(f=5), A(f=6), B(f=4)... Finds optimal path: S→A→B→G, cost=6 ✅ A* GRAPH Search: Closes S, then C (g=2), then D... Never re-opens C even if better path S→A→C (g=4) found. But g=4>g=2 anyway here, so result is same here. In general though: NOT guaranteed optimal with inconsistent h!
PITFALL #15 · HEURISTIC

Summer 2025 Tricky Heuristic: h5 and h6 Admissibility

// Summer 2025 Q3c: Given h1,h2,h3 admissible, h4 inadmissible: h5(n) = max(h1(n), h4(n)) + max(h2(n), h4(n)) − 2×h4(n) + h3(n) // Simplify: // Case 1: h4 ≥ h1 AND h4 ≥ h2: h5 = h4 + h4 - 2h4 + h3 = h3(n) → admissible! (h3 is admissible) // Case 2: h1 ≥ h4 AND h4 ≥ h2: h5 = h1 + h4 - 2h4 + h3 = h1 - h4 + h3 = h1 + h3 - h4 ≤ h1 + h3 ≤ h* + h* (if h*≥0, this could be >h*) Wait: h1 ≤ h* and h3 ≤ h*. So h1-h4+h3 ≤ h1+h3. But also h1-h4+h3 ≤ h1+h3 and h4≥0, so ≤ h1+h3. // Actually the key insight: max(h1,h4) + max(h2,h4) - 2h4 = max(h1-h4, 0) + max(h2-h4, 0) (after simplification) ≤ h1 + h2 (since max(h1-h4,0) ≤ max(h1,0) = h1) Adding h3: total ≤ h1+h2+h3, but all ≤ h* → NOT guaranteed admissible! // h6(n) = avg(h2,h3,h4) - h4/3 - 2h2/3 // = (h2+h3+h4)/3 - h4/3 - 2h2/3 // = h2/3 + h3/3 + h4/3 - h4/3 - 2h2/3 // = -h2/3 + h3/3 // = (h3 - h2)/3 // This can be negative! h(n) ≥ 0 required. Problematic. // Also if h2 > h3, negative value → 0 ≤ h(n) ≤ h* not guaranteed.

The systematic approach: Algebraically simplify the formula. If the simplified form involves only admissible heuristics with non-negative coefficients summing ≤ 1, it's admissible. If the formula can go negative, or if inadmissible h4 appears with a positive weight where it could dominate → NOT admissible.

Simulated Annealing Pitfalls

PITFALL #16 · SA

Wrong Sign for ΔE — Maximisation vs Minimisation!

ΔE is ALWAYS: E(neighbour) − E(current)

Problem TypeΔE > 0 meansΔE < 0 meansBad-move probability
MaximisationBetter (higher = good)Worse → "second chance"eΔE/T (ΔE is negative here)
MinimisationWorse (higher = bad)Better → always accepte−ΔE/T (ΔE is positive here)
⚠️

Watch out in exams: The question says "maximisation problem" (like Starlink Q4, Spring 2025) OR "minimisation" (like the Fall 2024 problem). These give DIFFERENT probability formulas. Using the wrong formula will give completely wrong answers.

PITFALL #17 · SA

Accept/Reject Decision — Which Way Does the Inequality Go?

▶ THE EXACT RULE
Compute probability: P = e^(ΔE/T) [for maximisation] Generate random number: RNG (given in problem) If RNG < P → ACCEPT the bad move (move there) If RNG ≥ P → REJECT the bad move (stay at current) // Common mistake: students write "RNG > P → accept". WRONG! // Lower RNG means "luckier" → easier to accept bad moves. // P is the THRESHOLD. RNG must be BELOW P to accept.
PITFALL #18 · SA

Good Moves Are ALWAYS Accepted — No Probability Check Needed!

❌ WRONG

"ΔE=+3 (good move, maximisation). P = e^(3/T) = e^0.3 = 1.35. Since RNG=0.5 < 1.35 → accepted." ← Using probability formula for a GOOD move!

✅ RIGHT

ΔE=+3 (good move, maximisation) → ALWAYS accept immediately. No probability calculation needed. No RNG comparison needed. Just move there and continue to next timestep.

PITFALL #19 · SA

Temperature Formulas — Calculate CAREFULLY

// Spring 2025: T = 100 - 0.2t t=0: T = 100 - 0 = 100 ✅ t=1: T = 100 - 0.2 = 99.8 ✅ t=4: T = 100 - 0.8 = 99.2 ✅ // Fall 2025: T = t - 2t² + 67 t=0: T = 0 - 0 + 67 = 67 ✅ t=1: T = 1 - 2(1) + 67 = 66 ✅ t=2: T = 2 - 2(4) + 67 = 2-8+67 = 61 ✅ t=3: T = 3 - 2(9) + 67 = 3-18+67 = 52 ✅ // Summer 2025: T = 8 - 2t t=0: T = 8. t=1: T=6. t=2: T=4. t=3: T=2. t=4: T=0! STOP here. // T=0 means no more bad moves accepted (pure hill climbing). // RNG formulas: // N = |sin(90 - 4t)| degrees t=0: N = |sin(90°)| = 1.0 t=1: N = |sin(86°)| ≈ 0.9976 t=4: N = |sin(74°)| ≈ 0.9613 // N = |cos(185 + 2.7t)| degrees t=0: N = |cos(185°)| = |cos(180+5)| = cos(5°) ≈ 0.9962 t=1: N = |cos(187.7°)| = cos(7.7°) ≈ 0.9910
💡

Exam speed tip: Always compute T(t) and N(t) in a table before starting calculations. Then do ΔE, P, compare for each timestep. Don't skip steps — each step is worth marks.

PITFALL #20 · SA

SA "Once Accepted, No Second Check"

Once SA accepts a neighbour (whether good or bad), you move there unconditionally. For the NEXT timestep, compute everything fresh from the NEW current state. Don't re-check whether you should have moved.

But note: After accepting a bad move (moving to worse state), the NEXT timestep still starts from the new (worse) state. The algorithm doesn't backtrack. This is by design — the bad move might have opened up a path to the global optimum!

Minimax & Alpha-Beta Pitfalls

PITFALL #21 · MINIMAX

Right-to-Left Traversal Changes Pruning Order!

❌ WRONG

"I'll just apply alpha-beta left-to-right regardless of what the question says."

✅ RIGHT

If the question says "right-to-left": evaluate the RIGHTMOST child first at every node. This sets α and β from right children first, and potentially prunes LEFT siblings. The final root value is the same, but different subtrees get pruned.

▶ EXAMPLE IMPACT
// Left-to-right: evaluate left children first, prune right // Right-to-left: evaluate right children first, prune left // Same minimax root value, different set of pruned nodes! // For "find ranges of a,b,c,d" questions (Summer 2025): // The unknown variables are leaves that SHOULD be pruned. // Find what value of 'a' causes α≥β at the moment 'a' would be evaluated. // That gives you the range constraint.
PITFALL #22 · MINIMAX

α and β Values — What They Mean and How They're Passed

WhenWho UpdatesFormulaPruning Check
At MAX node, after child returns vMAX updates αα = max(α, v)If α ≥ β → prune remaining children
At MIN node, after child returns vMIN updates ββ = min(β, v)If α ≥ β → prune remaining children
Going TO a childParent passesChild inherits parent's current (α, β)—
Returning FROM a childParent receives vUpdates own α or βChecks pruning

α starts at −∞ and can only INCREASE. β starts at +∞ and can only DECREASE. When they "cross" (α ≥ β), we're done with this subtree.

PITFALL #23 · MINIMAX

Minimax Value ≠ Minimax Move

▶ THE DISTINCTION

The minimax value is the guaranteed utility MAX can achieve (a number). The minimax move is the action MAX should take (which child to go to).

At the root, the move = go to the child whose minimax value equals the root's minimax value. If two children tie, pick left (or per tie-breaking rule).

Example: Root MAX value = 6. Left child (MIN-L) = 6. Right child (MIN-R) = 2. Minimax MOVE = go LEFT. Don't say "the answer is 6" — say "the optimal move is to go to MIN-L (left child), which guarantees utility 6".

PITFALL #24 · MINIMAX

Nodes Are Pruned — NOT "Not Evaluated" by the Algorithm

Pruned nodes are never evaluated (we save computation). But the key phrase in exam answers is: "this node is PRUNED because the pruning condition α ≥ β was met BEFORE reaching this node." You must state: (1) which node is pruned, (2) what α and β values caused it, (3) why this is a pruning condition.

PITFALL #25 · MINIMAX

Minimax Always Optimal? — The Nuanced Answer

✅ Optimal against RATIONAL opponent

If opponent plays perfectly (chooses minimum), minimax gives MAX the best guaranteed outcome.

❌ NOT optimal against irrational opponent

If opponent makes mistakes, a different strategy exploiting those mistakes could achieve higher utility than what minimax "pessimistically" guarantees.

Also NOT optimal when using depth-limited search + evaluation functions (approximate, not exact utility). This is the practical case in chess programs — they're not "optimal" in the theoretical sense.

Master Cheat Sheet — Everything on One Page

PEAS Template

Performance

What score does the agent maximise? Include: success criteria, efficiency (speed/cost), safety penalties, specific measurable outcomes.

Environment

Physical/digital world. Be specific: type of space, obstacles, other agents, dynamic elements, constraints (battery, time).

Actuators

Output devices: wheels/motors (move), robotic arm (manipulate), display/speaker (communicate), antenna (transmit).

Sensors

Input devices: camera (visual), microphone (audio), GPS (location), battery sensor, temperature sensor, obstacle detector.

Problem Formulation Template

State Space

All possible world configurations. Must capture everything needed to determine what happens next. Often a tuple: (location, items_remaining, battery).

Start State

Initial configuration. Must be a specific state, not vague. Example: (pharmacy, all_medicines_undelivered, full_battery).

Actions/Successor

What moves are possible + what they produce. Successor function: state × action → new_state. List all valid actions from any state.

Goal Test + Cost

Goal test: condition that defines success. Path cost: how to measure total cost of a solution path (moves, time, battery).

Simulated Annealing Quick Procedure

For each timestep t: 1. T = temperature_formula(t) ← compute T 2. N = |RNG_formula(t)| ← compute random number 3. Get selected neighbour (given) 4. ΔE = E(neighbour) - E(current) 5. If ΔE > 0 (good move, maximisation): → ALWAYS ACCEPT. Move. Done for this timestep. 6. If ΔE < 0 (bad move): → P = e^(ΔE/T) ← probability of accepting bad move → If N < P: ACCEPT (move to neighbour) → If N ≥ P: REJECT (stay at current)

Alpha-Beta Quick Procedure

Start: ALPHA-BETA(root, α=-∞, β=+∞) At each MAX node: For each child c (in given order): v = ALPHA-BETA(c, α, β) α = max(α, v) if α ≥ β: PRUNE! Break. return v At each MIN node: For each child c (in given order): v = ALPHA-BETA(c, α, β) β = min(β, v) if α ≥ β: PRUNE! Break. return v At leaf: return the given utility value.

All Formulas — Complete Reference

A* cost function:   f(n) = g(n) + h(n)
Admissibility:   h(n) ≤ h*(n)   for all nodes n
Consistency:   h(n) ≤ c(n, n') + h(n')   for all edges n→n'
SA bad move probability (MAXIMISATION):   P = e^(ΔE/T)   where ΔE = E(neighbor) - E(current)
SA bad move probability (MINIMISATION):   P = e^(-ΔE/T)   where ΔE = E(neighbor) - E(current)
Accept bad move if:   RNG < P     |   Reject if:   RNG ≥ P
Alpha-Beta prune condition:   α ≥ β
Minimax MAX:   max over all child values
Minimax MIN:   min over all child values
BFS time/space:   O(b^s)   |  DFS time: O(b^m), space: O(bm)   |  UCS: O(b^(C*/ε))
A* best case pruning:   O(b^(m/2))   — halves the effective depth
