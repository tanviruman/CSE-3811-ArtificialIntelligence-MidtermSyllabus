Search Heuristics — The Smart Shortcut

What is a Heuristic? A heuristic h(n) is a function that estimates how far node n is from the goal. It's domain-specific knowledge encoded as a number. It guides the search toward the goal without needing to explore everything.

From your notes: "Estimated a cost from a position to the goal." And: "UCS: source node is g=0. Heuristics: goal is h=0." — meaning at the goal, h=0 always!

Two Classic Heuristics

📐 Manhattan Distance (h₂)

Count of city blocks between current and goal. Only horizontal + vertical moves.

h(n) = |x₂ - x₁| + |y₂ - y₁|

Your notes: "0 ≤ h₂ ≤ h*". Used for grid problems, 8-puzzle (misplaced tiles + distance).

📏 Euclidean Distance (h₁)

Straight-line (as the crow flies) distance between current and goal.

h(n) = √[(x₂-x₁)² + (y₂-y₁)²]

Your notes: "0 ≤ h₁ ≤ h*". Used for Romania map (straight-line distance to Bucharest).

💡

Which is better? Manhattan ≥ Euclidean (when moves are grid-based). Higher heuristic (closer to true cost) = fewer nodes explored = faster. But must never EXCEED true cost (see Admissibility).

8-Puzzle Heuristics (From Your Notes)

Misplaced Tiles heuristic: h = count of tiles NOT in their correct position. Each misplaced tile adds 1 to h. This is admissible because each misplaced tile requires AT LEAST 1 move.

Manhattan Distance heuristic: h = sum of Manhattan distances of each tile from its goal position. Example: if tile 3 is 2 columns and 1 row away, it adds 3 to h. BETTER than misplaced tiles (higher, closer to true cost)!

Greedy Best-First Search

Idea: Always expand the node with the smallest h(n) — the one that seems closest to the goal. Uses ONLY the heuristic (no backward cost). Greedy = impatient!

f(n) = h(n) ← only forward estimate, no backward cost

Properties

Completeness

❌ NO — Can get trapped in loops.

Optimality

❌ NO — May take a longer path that looks close to goal.

Time

O(bm) worst case — can be much better in practice with good heuristic.

Space

O(bm) — stores all nodes in memory.

What can go wrong? Greedy can go directly to the "wrong goal" — a goal that looks close by heuristic but is actually a dead end or suboptimal path. Worst case: behaves like a badly-guided DFS!

💡

Greedy = good for quick-and-dirty solutions. Fast but not optimal. Use when speed matters more than solution quality.

⭐

A* Search — The Best of Both Worlds

A* combines UCS + Greedy: It uses both the actual cost so far AND the estimated remaining cost. This makes it both optimal AND informed (efficient).

f(n) = g(n) + h(n)

g(n) — Backward Cost

Actual cost from start to node n (the path cost so far). This is what UCS uses alone.

Your notes: "UCS − g(n)"

h(n) — Forward Cost

Estimated cost from node n to goal (heuristic). This is what Greedy uses alone.

Your notes: "Greedy − h(n)"

f(n) = estimated total cost of the cheapest solution passing through node n. A* expands nodes in order of increasing f(n) — always picks the node with the best estimated total cost!

Remember from your notes: "UCS = f(n) = g(n) + h⁰ (where h=0)" — if you set h=0, A* becomes exactly UCS! And if you set g=0, A* becomes Greedy!

Admissibility — Required for Optimality

Admissibility Rule: A heuristic h is admissible if it never overestimates the true cost to reach the goal. It's optimistic — it assumes things will go better than they actually will.

0 ≤ h(n) ≤ h*(n) for all nodes n

Where h*(n) = actual shortest path cost from n to goal.

From your notes (Class 5): "Range: 0 ≤ h(n) ≤ h*(n). h = assumption heuristic. h* = actual optimal cost (the true path cost)." And: "For nodes: h(S)=7, h*(S)=4 → 7 ≤ 4 → FALSE → Node S not admissible!"

How to Check Admissibility — Step by Step

1

For each node, compute the actual shortest path cost to goal (h*)

2

Compare: is h(n) ≤ h*(n) for EVERY node?

3

If ANY node fails (h(n) > h*(n)) → NOT admissible. Fix: reduce that h(n) to be ≤ h*(n)

Why Admissibility Ensures A* Tree Search is Optimal

If h never overestimates, then f(n) = g(n) + h(n) never overestimates the true cost of passing through n. So A* will always find and expand the true optimal goal first, before any suboptimal path.

Consistency (Monotonicity) — Required for A* Graph Search

Consistency Rule: A heuristic h is consistent if for every node n and every successor n' with edge cost c(n,n'):

h(n) ≤ c(n, n') + h(n')

This is the triangle inequality for heuristics. The estimated distance to goal from n should never exceed the cost to take one step + estimated distance from the next node.

From your notes (Class 6): "h(B) ≤ c + h(A)" (triangle inequality). And: "h(B) - h(A) ≤ c" — meaning the heuristic drop between 2 nodes is less than or equal to the actual cost between them.

Consistency vs Admissibility

Admissibility

h(n) ≤ h*(n) — never overestimates to goal.

Required for: A* Tree Search

Weaker condition. Every consistent h is admissible.

Consistency

h(n) ≤ c(n,n') + h(n') — triangle inequality holds.

Required for: A* Graph Search

Stronger condition. Guarantees f(n) never decreases along path.

KEY FACT: Consistency → Admissibility (but NOT the other way around!). A consistent heuristic is always admissible. An admissible heuristic is NOT necessarily consistent.

What Happens with Inconsistent Heuristic in Graph Search?

If h is admissible but NOT consistent, A* Graph Search may close a node via a non-optimal path (because the closed set prevents re-exploration). This means it might MISS the optimal solution!

Example from your notes (page 17): Graph S→A(1), S→B(1), B→C(2), A→C(1), C→G(3). h values: S=2, A=4, B=1, C=1, G=0. Checking S→A: h(S)=2 ≤ c(S,A)+h(A) = 1+4 = 5 ✓. But there may be an edge that violates this. When inconsistent → A* graph search found S→B→C→G instead of S→A→C→G (optimal). "Failed to find optimal path as it is not consistent."

A* Tree Search vs A* Graph Search

A* Tree Search

  • Uses an open set only
  • Allows duplicate/repeated states
  • May explore same state multiple times
  • Optimal if h is admissible
  • Slower (more nodes) but safe with admissible h

Your notes: "Open set · Duplicate allowed · Admissibility"

A* Graph Search

  • Uses a closed set (explored set)
  • Never expands the same node twice
  • More efficient — prunes explored states
  • Optimal if h is consistent
  • May fail with only admissible (not consistent) h

Your notes: "Closed set · No duplicate/seen · Easier for consistent heuristics"

Dominance — Which Heuristic is Better?

h₂ dominates h₁ if: h₂(n) ≥ h₁(n) for ALL nodes n (and both are admissible). A dominant heuristic explores fewer nodes because it provides tighter (more accurate) estimates.

0 ≤ h₁ ≤ h₂ ≤ h* → h₂ is BETTER (dominant over h₁)

From your notes: "0 ≤ hc ≤ ha ≤ h* → better." The semi-lattice shows: exact (h*) at top, then max(ha,hb), then individual ha, hb, then lower ones, then 0 at bottom. Higher = better heuristic.

Key Dominance Result

Best admissible combination: h_dom(n) = max(h₁(n), h₂(n)). Taking the max of admissible heuristics creates a new admissible heuristic that dominates both — the BEST of both worlds!

Why? If h₁ ≤ h* and h₂ ≤ h*, then max(h₁,h₂) ≤ h*. And max ≥ each individual, so it dominates both!

Heuristic Combination Rules — Exam Favourite!

Given h₁, h₂, h₃ are admissible (0 ≤ hᵢ ≤ h*) and h₄ is inadmissible (h₄ > h* possible). Determine which combinations are admissible:

New HeuristicAdmissible?Reason
h = max(h₁, h₂, h₃)✓ YESMax of admissible heuristics ≤ h* (all ≤ h*, so max ≤ h*). Also dominant!
h = min(h₁, h₂, h₄)✓ YESMin of any values ≤ min of admissible ones ≤ h*. Min can never exceed h*.
h = (h₁+h₂)/3✓ YESAverage of admissible ≤ average of h* = h*. (from your notes: "h6 = (h1+h2)/3 — Admissible")
h = h₁² + h₂ + h₃✗ NOh₁² can exceed h* even if h₁ ≤ h*. Squaring breaks the admissibility bound.
h = max(h₁, h₂, h₃, h₄)✗ NOh₄ can be > h*, and max could take that value.
h = 2h₁ + 3h₃✗ NOScaling admissible heuristics by factors > 1 can exceed h*. (your notes: "h7 = 2h₁+3h₃ — Not admissible")
h = (h₁+h₂+h₃+h₄)✗ NOIncludes h₄ which can overestimate. Sum can exceed h*.
h = max(min(h₁,h₃), h₄)✗ NOh₄ can be > h*, taking max of it with anything ≥ 0 can exceed h*.
🧠

Quick rules: max(admissible, admissible) = admissible ✓ | min(any including inadmissible) = admissible ✓ | average of admissibles = admissible ✓ | sum/product/scale involving inadmissible = RISKY ✗

The h5 and h6 Special Cases (Summer 2025 Midterm)

h5(n) = max(h₁,h₄) + max(h₂,h₄) - 2×h₄ + h₃

▶ ANALYSIS

Simplify: max(h₁,h₄) = h₁ if h₁≥h₄, else h₄. Similarly for max(h₂,h₄).

Case 1: h₁≥h₄ and h₂≥h₄: h5 = h₁ + h₂ - 2h₄ + h₃. If h₄ is large, this could reduce below 0 or still exceed h*. Generally admissible in this case.

Case 2: h₄>h₁ and h₄>h₂: h5 = h₄ + h₄ - 2h₄ + h₃ = h₃. Since h₃ admissible → h5 = h₃ ≤ h*. Admissible in this case!

Overall: h5 ≤ h₁ + h₂ + h₃ ≤ 3h*. Not guaranteed admissible in all cases. Conclusion: Not guaranteed admissible.
